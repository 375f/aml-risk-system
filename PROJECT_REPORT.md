## 1. Общее описание реализованной системы

AML Risk System — это веб-приложение на платформе Streamlit, которое помогает субъектам малого предпринимательства оценить риск блокировки расчётного счёта по критериям Федерального закона № 115-ФЗ «О противодействии легализации (отмыванию) доходов, полученных преступным путём, и финансированию терроризма».

Система решает прикладную задачу: ИП или бухгалтер загружает банковскую выписку или вводит ИНН контрагента — система автоматически вычисляет признаки риска, классифицирует уровень опасности и даёт объяснение, какие именно показатели вышли за нормативные пределы.

Система состоит из двух функциональных модулей, реализованных в рамках единого Streamlit-приложения:

- **Модуль 1 «Анализ выписки»** — принимает CSV или XLSX-файл банковской выписки, вычисляет 7 признаков риска на основе методических рекомендаций ЦБ РФ (18-МР и 19-МР), классифицирует уровень риска (low / medium / high) с помощью обученного RandomForestClassifier и объясняет результат через важность признаков.

- **Модуль 2 «Проверка контрагента»** — принимает ИНН, валидирует его по алгоритму контрольной суммы ФНС, запрашивает сведения с портала Rusprofile.ru, вычисляет балл риска контрагента по 5 критериям 115-ФЗ и выносит вердикт (safe / caution / high_risk).

Результаты обоих модулей сохраняются в PostgreSQL для ведения истории анализов. Предусмотрено кэширование данных о контрагентах с TTL 24 часа, чтобы избежать избыточных сетевых запросов.

**Поток данных — Модуль 1:**

1. Пользователь загружает файл через `st.file_uploader`.
2. `parse_statement(file)` определяет формат (Альфа-Банк / тип A / тип Б), выполняет fuzzy-маппинг колонок, возвращает `DataFrame[date, amount, type, description, counterparty, inn]`.
3. `compute_features(df, okved)` вычисляет 7 числовых признаков в диапазоне `[0, 1]`.
4. `predict(features)` формирует вектор `np.array` из 7 элементов, вызывает `pipeline.predict()` и `pipeline.predict_proba()`, возвращает `{risk_level, risk_proba, importances}`.
5. Результат записывается в таблицы `analysis_history` и `risk_factors` через `save_analysis()`.
6. `st.rerun()` перерисовывает страницу с секцией результата.

**Поток данных — Модуль 2:**

1. Пользователь вводит ИНН. `validate_inn(inn)` проверяет контрольную сумму в реальном времени.
2. При нажатии кнопки: проверяется кэш в таблице `contractors` (фильтр `expires_at > now()`).
3. Если кэша нет — `get_entity_by_inn(inn)` делает два HTTP-запроса к Rusprofile.ru и разбирает HTML.
4. `score_contractor(entity)` вычисляет балл 0–100 по 5 критериям.
5. Результат кэшируется с `expires_at = now() + 24h` через `save_contractor()`.
6. `st.rerun()` перерисовывает страницу с карточкой контрагента.

## 4. Модуль 1 — Анализ транзакций

### 4.1 Парсинг выписки (parser.py)

**Поддерживаемые форматы:**

| Формат | Описание | Банки |
|---|---|---|
| Формат А (split) | Отдельные колонки «Дебет» и «Кредит» | Сбербанк, ВТБ, большинство банков |
| Формат Б (amount+type) | Колонка «Сумма» + колонка «Тип операции» | Тинькофф и другие |
| Формат В (signed) | Колонка «Сумма» со знаком (минус=расход) | Ряд банков |
| Формат Альфа-Банка | Технические заголовки `d_c`, `sum_rur`, `text70` | Альфа-Банк (прямая выгрузка) |

Все форматы поддерживают кодировки `UTF-8`, `UTF-8-BOM`, `CP1251`, `Latin-1` и разделители `;`, `,`, `\t`, `|`.

**Автомаппинг колонок (fuzzy matching):**

Маппинг реализован через словарь синонимов `_ALIASES` (8 канонических имён → от 8 до 12 вариантов заголовков) и метрику `SequenceMatcher.ratio()`:

```python
def _score_header(header: str, aliases: list[str]) -> float:
    h = header.lower().strip()
    best = 0.0
    for alias in aliases:
        a = alias.lower().strip()
        if h == a:
            return 1.0
        if a in h or h in a:
            best = max(best, 0.90)  # substring match
            continue
        best = max(best, _similarity(h, a))  # SequenceMatcher
    return best
```

Жадное назначение: кандидаты сортируются по убыванию score, каждая сырая колонка и каждое каноническое имя используются не более одного раза. Порог cutoff = 0.60.

**Специализированный парсер Альфа-Банка:**

Обнаруживается по наличию сигнатурных колонок `frozenset({"d_c", "sum_rur", "text70"})`. Вторая строка файла (русские описания колонок) пропускается по признаку: значение `sum_rur` в строке 0 не является числом. Контрагент и ИНН определяются направленно: при `d_c == "D"` (расход) берётся `pol_name/pol_inn`, при `d_c == "C"` (приход) — `plat_name/plat_inn`.

**Обрабатываемые исключения:**

- `ParseError` — пустой файл, менее 2 колонок, не удалось прочитать CSV
- `ColumnMappingError(ParseError)` — не найдена колонка с датой, с назначением платежа или с суммами
- Все прочие исключения перехватываются в `module1/ui.py` и отображаются через `st.error()`

**Поддерживаемые форматы дат:** `%d.%m.%Y`, `%d/%m/%Y`, `%Y-%m-%d`, `%d.%m.%Y %H:%M`, `%d.%m.%Y %H:%M:%S`, `%Y-%m-%dT%H:%M:%S`, `%d/%m/%Y %H:%M:%S`, `%Y%m%d`. При неуспехе всех форматов вызывается `pd.to_datetime(..., errors="coerce")`.

### 4.2 Вычисление признаков риска (features.py)

Функция `compute_features(df, okved)` принимает стандартизованный DataFrame и опциональный код ОКВЭД. Возвращает словарь из 7 признаков, каждый в диапазоне `[0.0, 1.0]`.

**Признак 1: Доля наличных операций (`cash_ratio`)**

- **Формула:** `cash_ratio = Σ(суммы по ключевым словам) / Σ(все дебетовые суммы)`
- **Ключевые слова:** «снятие наличных», «выдача наличных», «банкомат», «касса», «наличные», «обналичивание», «выдача нал»
- **Порог:** `>= 0.30` (30%)
- **Источник:** Методические рекомендации ЦБ РФ 19-МР от 21.07.2017

**Признак 2: Налоговая нагрузка (`tax_ratio`)**

- **Формула:** `tax_ratio = Σ(налоговые платежи) / Σ(все дебетовые суммы)`
- **Ключевые слова:** «ндс», «налог», «ифнс», «фнс», «пфр», «взносы», «фсс», «ффомс», «есн», «страховые взносы», «бюджет»
- **Порог:** `< 0.009` (менее 0.9%) — направление риска «ниже = хуже»
- **Источник:** Методические рекомендации ЦБ РФ 18-МР от 21.07.2017

**Признак 3: Транзитный характер операций (`transit_ratio`)**

- **Формула:** Rolling-window алгоритм: для каждого дня с кредитовыми поступлениями вычисляется, какая доля средств ушла в течение последующих 3 дней:
  ```
  transit_sum += min(credit_day, debit_in_window_3days)
  transit_ratio = transit_sum / total_credits
  ```
- **Порог:** `> 0.50` (более 50%)
- **Источник:** 19-МР ЦБ РФ от 21.07.2017

**Признак 4: Несоответствие ОКВЭД (`okved_mismatch`)**

- **Формула:** `okved_mismatch = 1 - (платежи с совпадающим назначением / все дебетовые платежи)`
- **Алгоритм:** По первым двум цифрам кода ОКВЭД выбирается словарь ожидаемых ключевых слов (`_OKVED_KEYWORDS`, покрывает 15 разделов ОКВЭД). Если ОКВЭД не передан или не распознан — возвращается `0.0`.
- **Порог:** `> 0.60` (более 60% платежей не соответствуют)
- **Источник:** Экспертное правило

**Признак 5: Средняя сумма транзакции (`avg_tx_norm`)**

- **Формула:** `avg_tx_norm = min(avg_credit / 600_000, 1.0)`
- **Нормировка:** Порог обязательного контроля 600 000 руб. по 115-ФЗ ст. 6. Значение < 1.0 указывает на возможное дробление операций (structuring).
- **Порог:** `< 1.0` — направление риска «ниже = хуже»
- **Источник:** 115-ФЗ, статья 6

**Признак 6: Концентрация контрагентов (`counterparty_concentration`)**

- **Формула:** `concentration = 1 - (уникальных контрагентов / кол-во транзакций)`
- Значение 0.0 — каждая транзакция с уникальным контрагентом (норма). Значение → 1.0 — все деньги уходят одному получателю.
- **Порог:** `> 0.80` (более 80%)
- **Источник:** 19-МР ЦБ РФ от 21.07.2017

**Признак 7: Доля переводов физлицам (`fl_ratio`)**

- **Формула:** `fl_ratio = Σ(суммы физлицам) / Σ(все дебетовые суммы)`
- **Обнаружение физлица** — по трём критериям (любой из них достаточен):
  1. ИНН в строке содержит 12 цифр (ИП или физическое лицо)
  2. Назначение платежа содержит ключевые слова: «зарплата», «дивиденд», «физическое лицо», «самозанятый» и др.
  3. Поле контрагента соответствует ФИО-паттерну (`Иванов И.И.` или `Иванов Иван Иванович`) и не содержит маркеров юрлица («ооо», «банк», «ифнс» и др.)
- **Порог:** `> 0.30` (более 30%)
- **Источник:** 19-МР ЦБ РФ от 21.07.2017

### 4.3 ML-модель (classifier.py + ml/)

**Алгоритм:** `RandomForestClassifier` с предобработкой `StandardScaler` в виде sklearn `Pipeline`.

**Параметры лучшей модели** (результат GridSearchCV):
- `n_estimators = 100`
- `max_depth = 5`
- `class_weight = "balanced"` — компенсирует несбалансированность классов

**GridSearchCV:** 9 комбинаций (3 значения `n_estimators` × 3 значения `max_depth`), 5-fold StratifiedKFold, метрика оптимизации — `f1_macro`.

**Датасет:** Синтетический, 1000 строк. Распределение классов: 60% low (600), 25% medium (250), 15% high (150). Параметры нормальных распределений подобраны по нормативным порогам ЦБ РФ. Для реалистичной межпризнаковой корреляции применён латентный фактор интенсивности риска:

```python
col += direction * intensity * (std * _LATENT_STRENGTH)
# _LATENT_STRENGTH = 0.40 — доля от std признака
# intensity = rng.normal(0.0, 1.0, n)  # Z ~ N(0,1)
```

**Метрики на тест-выборке (20%, т.е. 200 строк):**

*Примечание:* Метрики 1.0 объясняются синтетическим характером данных — классы хорошо разделимы, т.к. генерировались с разнесёнными центрами распределений. На реальных данных ожидается снижение точности.

**Объяснение через `feature_importances_`:**

В `classifier.py` после вызова `model.predict()` извлекается атрибут `rf.feature_importances_` из шага `"clf"` конвейера:

```python
rf = model.named_steps["clf"]
importances: dict[str, float] = {
    k: float(v) for k, v in zip(FEATURE_KEYS, rf.feature_importances_)
}
```

В UI эти значения отображаются в виде горизонтальной гистограммы (`st.bar_chart`) и включаются в таблицу `risk_factors` в поле `importance`.

## 5. Модуль 2 — Проверка контрагента

### 5.1 Валидация ИНН (validator.py)

Реализована полная проверка по алгоритму контрольной суммы, утверждённому приказом ФНС России.

**10-значный ИНН (юридическое лицо):**

Весовые коэффициенты: `_W10 = [2, 4, 10, 3, 5, 9, 4, 6, 8]`

```
n = (Σ _W10[i] * d[i] for i in 0..8) % 11 % 10
valid: n == d[9]
```

**12-значный ИНН (ИП / физическое лицо):**

Две контрольных цифры:
- `_W11 = [7, 2, 4, 10, 3, 5, 9, 4, 6, 8]` → `n11 = Σ % 11 % 10`, проверяем `n11 == d[10]`
- `_W12 = [3, 7, 2, 4, 10, 3, 5, 9, 4, 6, 8]` → `n12 = Σ % 11 % 10`, проверяем `n12 == d[11]`

**Функции:**
- `validate_inn(inn: str) -> bool` — полная проверка; `False` если строка не состоит из цифр или длина не 10/12
- `get_inn_type(inn: str) -> str` — возвращает `"ul"` (10 цифр) или `"ip"` (12 цифр)

### 5.2 Интеграция с Rusprofile (rusprofile_api.py)

**Отклонение от CLAUDE.md:** Вместо `api.egrul.nalog.ru` (требует токена) реализован HTTP-парсер Rusprofile.ru.

**Алгоритм (2 HTTP-запроса):**

```python
_SEARCH_URL = "https://www.rusprofile.ru/search?query={query}"

# Шаг 1: поисковая страница
resp = requests.get(_SEARCH_URL.format(query=inn), headers=_HEADERS, timeout=30)
canonical_url = _extract_canonical(resp.text)
# Если ИНН не найден — canonical указывает на /search или /find

# Шаг 2: страница компании (версия для печати)
resp2 = requests.get(canonical_url + "?print=1", headers=_HEADERS, timeout=30)
kv = _parse_html_text(resp2.text, inn)
```

**Заголовки запросов:** User-Agent Firefox 62, Accept text/html, Accept-Language ru-RU.

**Определение «ИНН не найден»:** canonical URL содержит `/search` или `/find`, либо отсутствует поле `orgShortName` и `orgFullName` после парсинга.

**Извлекаемые поля из HTML:**

| Поле HTML | Внутренний ключ | Тип |
|---|---|---|
| `class="company-name"` → `<a>` | `orgShortName` | str |
| «Полное наименование» | `orgFullName` | str |
| «Статус» | `orgStatus` | str |
| «Юридический адрес» | `orgAddress` | str |
| «Уставный капитал» | `commonBaseFunds` | str (числовая строка) |
| «ОГРН» | `commonOGRN` | str |
| «ОГРНИП» | `commonOGRNIP` | str |
| «Дата регистрации» | `commonRegDate` | str (конвертируется из рус. формата) |
| «ФИО» / «Имя» | `orgFullName` (fallback) | str |

**Нормализация статуса** (stem-matching, порядок важен):

| Подстрока в русском тексте | Код |
|---|---|
| «в процессе ликвидации» / «ликвидируется» / «ликвидаци» | `liquidating` |
| «прекращен» / «недействующ» / «ликвидир» / «исключ» | `liquidated` |
| «реорганиза» | `reorganizing` |
| «действующ» | `active` |

**Обработка ошибок:**
- `requests.Timeout` → «Rusprofile не ответил за 30 секунд»
- `requests.HTTPError` → «Ошибка HTTP: {status_code}»
- `requests.RequestException` → «Ошибка сети: {e}»
- ИНН не найден → «ИНН {inn} не найден на Rusprofile»

**Fallback на кэш:** В `module2/ui.py._run_check()` перед обращением к Rusprofile вызывается `get_contractor_cache(session, inn)`. Если кэш не истёк — сетевой запрос не делается.

### 5.3 Скоринг контрагента (scorer.py)

**Полная таблица критериев:**

| Критерий | Условие | Баллы | Описание в triggered |
|---|---|---|---|
| Статус | `== "liquidated"` | +50 | «Организация ликвидирована» |
| Статус | `== "liquidating"` | +40 | «Организация в процессе ликвидации» |
| Срок регистрации | `age_months < 3` | +30 | «Зарегистрирована менее 3 месяцев назад» |
| Срок регистрации | `age < 6` | +20 | «Зарегистрирована менее 6 месяцев назад» |
| Срок регистрации | `age < 12` | +10 | «Зарегистрирована менее года назад» |
| Массовый адрес | `mass_address == True` | +20 | «Массовый адрес регистрации» |
| Массовый руководитель | `mass_director == True` | +20 | «Массовый руководитель» |
| Уставный капитал | `capital is not None and <= 10 000` | +10 | «Минимальный уставный капитал (10 000 руб.)» |

Максимальный балл ограничен 100 (`score = min(score, 100)`).

**Пороги вердиктов:**
- `score <= 25` → `"safe"` (Безопасно) — зелёный
- `score <= 60` → `"caution"` (Осторожно) — жёлтый
- `score > 60` → `"high_risk"` (Высокий риск) — красный

**Пример расчёта для ликвидируемой молодой организации:**

```
Статус: "liquidating"     → +40
Возраст: 4 месяца         → +20 (< 6 мес.)
Массовый адрес: нет       → +0
Массовый директор: нет    → +0
Уставный капитал: 10 000  → +10

Итого: 70 → "high_risk"
triggered: ["Организация в процессе ликвидации", "Зарегистрирована менее 6 месяцев назад",
            "Минимальный уставный капитал (10 000 руб.)"]
```

**Ограничение:** `mass_address` и `mass_director` всегда возвращаются как `False` из `rusprofile_api.py`, поскольку Rusprofile.ru не предоставляет эти данные в открытом виде. Критерии 3 и 4 фактически не работают при текущем источнике данных.

| Колонка | Тип | Описание |
|---|---|---|
| `id` | SERIAL PRIMARY KEY | Автоинкремент |

| `expires_at` | TIMESTAMP | TTL: время истечения кэша |
| `raw_json` | JSONB | Полный ответ от Rusprofile |

Индексы: `idx_contractors_inn (inn)`, `idx_contractors_expires (expires_at)`.

### Сохранение данных

- **При каждом анализе выписки:** `save_analysis()` создаёт запись в `analysis_history` и N записей в `risk_factors` (по одной на каждый из 7 признаков). При ошибке БД — исключение перехватывается молча, результат всё равно показывается пользователю.
- **При каждой проверке контрагента:** `save_contractor()` создаёт или обновляет (по уникальному `inn`) запись в `contractors`. Поле `checked_at` обновляется при каждом обновлении. Ошибки БД также перехватываются молча.

### Кэш контрагентов (TTL)

TTL задаётся переменной окружения `CONTRACTOR_CACHE_TTL_HOURS` (по умолчанию 24 часа, значение из `config.py`):

```python
expires = datetime.now(timezone.utc) + timedelta(hours=CONTRACTOR_CACHE_TTL_HOURS)
```

Проверка кэша:
```python
stmt = select(Contractor).where(Contractor.inn == inn).where(Contractor.expires_at > now)
```

Если `expires_at` истёк — запись остаётся в базе, но функция `get_contractor_cache()` возвращает `None`, и будет сделан новый сетевой запрос.

### ORM-модели и CRUD

- **`db/connection.py`:** `create_engine()` с `pool_pre_ping=True`, `pool_size=5`, `max_overflow=10`. `SessionLocal = sessionmaker(autocommit=False, autoflush=False, expire_on_commit=False)`.
- **`db/models.py`:** Три ORM-класса с typed mapped columns (SQLAlchemy 2.0 стиль). Связь `AnalysisHistory` ↔ `RiskFactor` через `relationship(..., cascade="all, delete-orphan")`.
- **`db/crud.py`:** Функция `_json_safe()` рекурсивно конвертирует объекты `date`/`datetime` в ISO-строки перед сохранением в JSONB (это необходимо, т.к. psycopg2 не умеет сериализовывать Python `date` в JSON автоматически).

## 8. Что реализовано, что не реализовано

### Полностью готово и работает

- **Парсинг выписок** — форматы А (split), Б (amount+type), В (signed), Альфа-Банк. Поддержка 4 кодировок и 4 разделителей. Автомаппинг колонок через fuzzy-matching.
- **Все 7 признаков риска** — реализованы строго по нормативам 18-МР, 19-МР, 115-ФЗ. Признак 3 (транзит) реализован rolling-window алгоритмом, а не примитивным сравнением дат.
- **ML-классификатор** — обучен, сохранён в `ml/model.joblib`. Pipeline (StandardScaler + RandomForest) загружается при первом обращении и кэшируется в памяти. Объяснение через `feature_importances_`.
- **Генерация и обучение датасета** — `ml/dataset.py` и `ml/train.py` полностью автономны. Синтетические данные с межпризнаковой корреляцией через латентный фактор.
- **Валидация ИНН** — полный алгоритм контрольной суммы ФНС для 10- и 12-значных ИНН.
- **Интеграция с Rusprofile** — два HTTP-запроса, разбор HTML без внешних библиотек, нормализация статусов через stem-matching, обработка всех сетевых ошибок.
- **Скоринг контрагента** — 5 критериев, балл 0–100, три вердикта. Все пороги вынесены в `config.py`.
- **База данных** — три таблицы, ORM-модели (SQLAlchemy 2.0 Mapped), CRUD-функции, TTL-кэш контрагентов, JSON-сериализация дат.
- **UI обоих модулей** — полноценный интерфейс с загрузкой файлов, ввода ИНН, отображения результатов, истории анализов. Тёмная тема Т-Банка.
- **Тесты** — `tests/test_parser.py`, `tests/test_features.py`, `tests/test_validator.py`, `tests/test_scorer.py`, `tests/test_rusprofile_api.py`.

### Реализовано частично

- **Признак `okved_mismatch`** — работает только при передаче кода ОКВЭД пользователем. Если поле ОКВЭД пустое — возвращается 0.0, признак не сработает. Покрыто 15 из 88 разделов ОКВЭД.
- **Массовый адрес и массовый руководитель** — поля `mass_address` и `mass_director` всегда `False`, т.к. Rusprofile.ru не предоставляет эту информацию. Критерии 3 и 4 скоринга фактически не задействованы.
- **Метрики ML-модели** — Accuracy/F1 = 1.0 на синтетических данных, что не отражает качество на реальных выписках. Реальная точность потребует разметки реальных кейсов.

### Не реализовано (заглушки или отсутствует)

- **API ФНС (`api.egrul.nalog.ru`)** — заменён парсером Rusprofile. Файл `module2/fns_api.py` является заглушкой-реэкспортом: `from module2.rusprofile_api import *`.
- **Аутентификация пользователей** — намеренно исключена по ТЗ.
- **Хранение файлов выписок** — намеренно исключено (хранятся только вычисленные признаки).
- **Автоматическая очистка истёкшего кэша** — записи с истёкшим `expires_at` остаются в базе. Фильтрация происходит только при запросе, физического удаления нет.
- **Автоматическое обновление модели** — `ml/model.joblib` статичен, дообучение на новых данных не предусмотрено.

## 9. Инструкция по запуску

### Предварительные требования

- Python 3.11+
- PostgreSQL 14+ с пустой базой данных `aml_db`
- Доступ к интернету (для Rusprofile.ru)

### Шаг 1: Клонирование и установка зависимостей

```bash
cd aml_system
python -m venv venv
# Windows:
venv\Scripts\activate
# Linux/macOS:
source venv/bin/activate

pip install -r requirements.txt
```

### Шаг 2: Настройка переменных окружения

Создать файл `.env` в папке `aml_system/`:

```env
DATABASE_URL=postgresql://postgres:password@localhost:5432/aml_db
RUSPROFILE_TIMEOUT=30
CONTRACTOR_CACHE_TTL_HOURS=24
MODEL_PATH=ml/model.joblib
```

### Шаг 3: Инициализация базы данных

```bash
psql -U postgres -c "CREATE DATABASE aml_db;"
psql -U postgres -d aml_db -f db/migrations/init.sql
```

Альтернатива через Python:

```bash
python db/init_db.py
```

### Шаг 4: Генерация данных и обучение модели

Если файл `ml/model.joblib` отсутствует:

```bash
python ml/dataset.py   # → генерирует ml/data.csv (1000 строк)
python ml/train.py     # → обучает и сохраняет ml/model.joblib
```

Для проверки качества модели:

```bash
python ml/evaluate.py
```

### Шаг 5: Запуск приложения

```bash
streamlit run app.py
```

Приложение будет доступно по адресу `http://localhost:8501`.

## 10. Возможные улучшения

Следующие направления доработки могут составить раздел «Дальнейшие перспективы» в дипломной работе:

**1. Реальные обучающие данные и дообучение модели.** Синтетический датасет не отражает реальное распределение признаков. Накопление размеченных реальных кейсов (решения банка о блокировке / не-блокировке) позволит обучить модель, применимую на практике. При этом целесообразно применение SHAP-значений вместо `feature_importances_` для объяснения конкретного предсказания.

**2. Проверка по официальному API ФНС.** После получения токена партнёра системы «Прозрачный бизнес» (api.egrul.nalog.ru) можно заменить Rusprofile-парсер на официальный источник. Это даст доступ к полям `mass_address` и `mass_director`, которые сейчас всегда `False`.

**3. Расширение ОКВЭД-словаря.** Текущий словарь `_OKVED_KEYWORDS` покрывает 15 из 88 разделов. Расширение до полного охвата улучшит качество признака `okved_mismatch`.

**4. Мониторинг и алерты.** Интеграция с системой уведомлений (email, Telegram) при обнаружении высокого риска. Периодическое переобнаружение контрагентов из истории (фоновая задача).

**5. Многопользовательский режим и аутентификация.** Разграничение доступа к данным разных ИП/бухгалтеров через механизм сессий Streamlit или вынесение в отдельный API-слой.

**6. Поддержка дополнительных форматов выписок.** XML-формат (ISO 20022), OFX, выписки в PDF (с распознаванием через OCR).

**7. Временно́й анализ тренда.** Сравнение признаков текущего периода с предыдущими анализами того же счёта для выявления ухудшения показателей во времени.

**8. Выгрузка отчётов.** Генерация PDF-заключений по результатам анализа для предъявления в банк или аудиторам.
