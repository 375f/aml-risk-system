## 1. Что это за программа и зачем она нужна

### Исходная идея

Дипломная работа студента 4 курса КНИТУ-КАИ, специальность 09.03.04 «Программная инженерия».
Тема: «Разработка интеллектуальной системы раннего предупреждения и предиктивной аналитики рисков блокировки банковских счетов».

В России банки обязаны по 115-ФЗ («О противодействии легализации доходов, полученных преступным путём») блокировать счета клиентов при подозрительной активности. ЦБ РФ опубликовал методические рекомендации (18-МР и 19-МР) с конкретными числовыми порогами. Малый бизнес (ИП, ООО) часто не знает об этих правилах и получает блокировку неожиданно.

**Цель системы** — дать ИП инструмент самопроверки:
1. Загрузить банковскую выписку → получить анализ риска по критериям ЦБ РФ
2. Ввести ИНН контрагента → получить оценку надёжности партнёра

### Чем система НЕ является

- Не банковская AML-система (не имеет доступа к реальным данным банка)
- Не юридически значимый инструмент (носит информационный характер)
- Не веб-сервис (нет REST API, нет аутентификации)
- Не нейросеть (строго Random Forest по требованиям диплома)

## 2. Архитектура — один процесс, два модуля

Система реализована как **единое Streamlit-приложение** без разделения на frontend/backend.
Streamlit сам является и UI-фреймворком, и сервером. Всё выполняется в одном Python-процессе.

```
streamlit run app.py
        │
        ▼
    app.py  ──── глобальные CSS-стили (тёмная тема Т-Банка)
        │         ──── боковая панель (сайдбар с нормативной базой)
        │         ──── вкладка 1 → module1/ui.py → render()
        │         ──── вкладка 2 → module2/ui.py → render()
        │
   ┌────┴────────────────────────────────────┐
   │                                         │
МОДУЛЬ 1                                 МОДУЛЬ 2
Анализ выписки                           Проверка контрагента
   │                                         │
parser.py        ← CSV/XLSX               validator.py  ← ИНН (строка)
features.py      ← DataFrame             rusprofile_api.py ← HTTP
classifier.py    ← dict признаков        scorer.py     ← dict данных
   │                                         │
   └──────────────── db/ ──────────────────┘
                      │
               connection.py (SQLAlchemy engine)
               models.py     (ORM-классы)
               crud.py       (save/get функции)
                      │
               PostgreSQL (aml_db)
```

### Почему Streamlit, а не Flask/FastAPI+React

Требование диплома + главная причина: Streamlit позволяет реализовать полноценный UI
без написания JavaScript. Весь интерфейс — Python-код. Нет SPA, нет API-контракта,
нет CORS, нет JWT. Для учебного проекта это правильный выбор.

**Нюанс**: Streamlit перезапускает весь Python-скрипт сверху вниз при каждом взаимодействии
пользователя (клик на кнопку, ввод в поле). Сохранение состояния между перезапусками —
через `st.session_state` (словарь, живущий на время сессии браузера).

## 5. config.py — единый источник правды

Все числовые пороги и константы хранятся в `config.py`. Ни один модуль не хардкодит
пороговые значения внутри себя — только импортирует из config.

```python
# Пороги Модуля 1 (нормативы ЦБ РФ)
CASH_RATIO_THRESHOLD    = 0.30   # 19-МР: доля наличных >= 30% → риск
TAX_RATIO_THRESHOLD     = 0.009  # 18-МР: налоговая нагрузка < 0.9% → риск
TRANSIT_RATIO_THRESHOLD = 0.50   # > 50% поступлений списываются за 3 дня
OKVED_MISMATCH_THRESHOLD = 0.60  # > 60% платежей не соответствуют ОКВЭД
FL_RATIO_THRESHOLD      = 0.30   # > 30% расходов — переводы физлицам
CONCENTRATION_THRESHOLD = 0.80   # > 80% расходов — одному контрагенту
# ... и ещё 4 экспертных правила
```

`config.py` также содержит `OKVED_KEYWORDS` — словарь из 114 кодов ОКВЭД (формат `"62.01": ["разработка", ...]`),
который импортируется в `features.py` для проверки соответствия транзакций виду деятельности.

## 6. Модуль 1 — Анализ выписки

### 6.1. parser.py — чтение файла

**Задача**: принять CSV или XLSX любого российского банка (Сбер, Альфа, Тинькофф, ВТБ...)
и вернуть стандартный DataFrame с колонками: `date, amount, type, description, counterparty, inn`.

**Проблема**: у каждого банка свои заголовки. «Дата операции», «Дата платежа», «date», «Дата документа» — всё это одно и то же поле.

**Решение**: словарь `_ALIASES` (canonical → список синонимов) + функция `_score_header()`,
которая считает схожесть через `SequenceMatcher`. Жадное назначение (greedy assignment):
идём от лучшего совпадения к худшему, каждую колонку назначаем только один раз.

```python
# Три формата выписок:
# Формат А (Сбербанк): отдельные колонки Дебет / Кредит
# Формат Б (большинство): Сумма + Тип операции
# Формат Альфа: d_c / sum_rur / text70 (специализированный парсер)
```

**Нюансы**:
- `_detect_header_row()` сканирует первые 15 строк XLSX в поисках строки-заголовка
  (некоторые банки добавляют строки с реквизитами перед таблицей транзакций)
- `_read_csv()` перебирает комбинации кодировок (utf-8, cp1251, latin-1) и разделителей (`;`, `,`, `\t`)
- Файл **не сохраняется на диск** — только bytes в памяти через `io.BytesIO`
- `df.head(10)` используется только в превью, анализ всегда получает полный df

### 6.2. features.py — 10 признаков риска

Центральный модуль системы. Принимает полный DataFrame, возвращает dict из 10 чисел в [0, 1].

#### Признак 1 — cash_ratio (19-МР ЦБ РФ)
Доля дебетовых (исходящих) транзакций с ключевыми словами «снятие наличных», «банкомат», «касса».
Порог: ≥ 30%.

#### Признак 2 — tax_ratio (18-МР ЦБ РФ)
Доля дебетовых транзакций с ключевыми словами «ндс», «налог», «ифнс», «пфр».
**Инверсный**: риск, если значение < 0.9% (слишком мало налогов).

#### Признак 3 — transit_ratio (19-МР ЦБ РФ)
Транзитная схема: деньги поступают и быстро уходят.
Алгоритм: для каждого дня с кредитовыми поступлениями считаем, сколько из них списалось
в течение следующих 3 дней (rolling window). `transit_sum / total_credit`.
Порог: > 50%.

#### Признак 4 — okved_mismatch
Долчасть входящих (кредитовых) транзакций, назначение которых не содержит ключевых слов
из словаря `OKVED_KEYWORDS` для заявленных кодов ОКВЭД.
Если ОКВЭД не указан — значение 0.0 (нет ложных срабатываний).
Подробнее: функция `check_okved_compliance()`.

#### Признак 5 — avg_tx_norm (115-ФЗ, ст. 6)
Средняя сумма входящей транзакции, нормированная к порогу обязательного контроля (600 000 руб.).
`min(avg_credit / 600_000, 1.0)`. **Инверсный**: низкое значение = дробление платежей.

#### Признак 6 — counterparty_concentration (19-МР ЦБ РФ)
`1 - (unique_counterparties / total_debit_transactions)`.
0 = каждая операция с разным контрагентом (норма). 1 = все деньги одному (транзит).
Порог: > 80%.

#### Признак 7 — fl_ratio (19-МР ЦБ РФ)
Доля расходов, уходящих физическим лицам. Физлицо определяется по трём критериям:
1. 12-значный ИНН (физлицо или ИП без статуса юрлица)
2. Ключевые слова в назначении («зарплата», «дивиденды», «самозанятый»...)
3. ФИО-паттерн в поле контрагента при отсутствии маркеров юрлица («ООО», «ОАО», «ИНН»...)

#### Признак 8 — weekend_ratio (экспертное правило)
Доля операций в субботу и воскресенье. Для легального бизнеса расчёты между юрлицами
происходят в рабочие дни. Высокая активность в выходные нетипична. Порог: > 30%.

#### Признак 9 — round_amount_ratio (FATF Typologies — структурирование)
Доля транзакций с суммами, кратными 10 000 руб. Намеренное использование круглых сумм —
классический признак структурирования: дробление крупных сумм для обхода порогов контроля.
Порог: > 60%.

#### Признак 10 — tx_frequency_norm (экспертное правило)
Нормализованная частота транзакций: `tx_per_day / 30`. При 15+ транзакций в день
при малых суммах — признак дробления. Порог: > 0.50 (более 15 транзакций в день).

### 6.4. UI — module1/ui.py

Три секции, вызываемые из `render()`:

**_section_upload()** — загрузка файла:
- `st.file_uploader` для CSV/XLSX
- Поле ввода кодов ОКВЭД (через запятую → `list[str]`)
- Кэш parsed DataFrame в `st.session_state[f"parsed_{name}_{size}"]`
  (исключает повторный парсинг при повторном нажатии кнопок)
- 4 метрики: транзакций, исходящие (ruб.), входящие (руб.), период
- Превью: первые 10 строк (не обрезает анализ — только отображение)

**_run_analysis()** — вызывает pipeline:
1. `compute_features(df, okved_codes)` → dict из 10 признаков
2. `describe_features(features)` → список с метаданными для UI
3. `predict(features)` → risk_level, proba, importances
4. `check_okved_compliance(df, okved_codes)` → okved_report
5. `save_analysis(session, {...})` → запись в PostgreSQL (в try/except — БД не обязательна)
6. `st.session_state["analysis_result"] = {...}` + `st.rerun()`

**_section_result()** — результат анализа:
- Карточка уровня риска с цветной левой полосой (`border-left: 4px solid {color}`)
- Цвета: 🟢 #00C853 (low), 🟡 #FFE000 (medium), 🔴 #FF3B30 (high)
- Bar chart важности признаков (Streamlit native `st.bar_chart`)
- Таблица признаков: значение, порог, статус (⚠ Риск / ✓ Норма), источник норматива
- Карточки сработавших факторов с `border-left: 3px solid #FFE000`
- Блок соответствия ОКВЭД (`_section_okved_compliance`)

**_section_history()** — история последних 50 анализов из PostgreSQL:
- Фильтры: по дате и уровню риска
- Если БД недоступна — `st.warning()`, приложение продолжает работать

## 7. Модуль 2 — Проверка контрагента

### 7.1. validator.py — валидация ИНН

ИНН юрлица (10 цифр) и ИП/физлица (12 цифр) имеют контрольные суммы.
Алгоритм: взвешенная сумма цифр по фиксированным весам → контрольная цифра.
Если контрольная сумма не совпадает — ИНН фиктивный.

```python
_WEIGHTS_10 = [2, 4, 10, 3, 5, 9, 4, 6, 8]   # для 10-значного
_WEIGHTS_12_1 = [7, 2, 4, 10, 3, 5, 9, 4, 6, 8]
_WEIGHTS_12_2 = [3, 7, 2, 4, 10, 3, 5, 9, 4, 6, 8]
```

### 7.2. rusprofile_api.py — получение данных

**Почему Rusprofile, а не официальный API ФНС**: официальный API ЕГРЮЛ (`api.egrul.nalog.ru`)
технически нестабилен, требует разбора сложного XML, нередко возвращает пустые ответы.
Rusprofile — агрегатор данных ЕГРЮЛ/ЕГРИП с удобным HTML-интерфейсом.

**Алгоритм (два HTTP-запроса)**:
1. `GET rusprofile.ru/search?query={ИНН}` — поиск
2. Извлечь `<link rel="canonical">` → URL страницы компании
3. `GET {canonical_url}?print=1` — версия для печати (содержит все реквизиты)
4. Разобрать HTML: таблицы `dt-text` → поле → значение

Возвращает стандартный dict:
```python
{
    "name": str | None,
    "status": "active" | "liquidated" | "liquidating" | ...,
    "reg_date": date | None,
    "address": str | None,
    "ogrn": str | None,
    "entity_type": "ul" | "ip",
    "mass_address": False,    # Rusprofile не предоставляет
    "mass_director": False,   # Rusprofile не предоставляет
    "capital": float | None,
}
```

**Нюанс**: `mass_address` и `mass_director` всегда `False` — Rusprofile не публикует эти данные.
В официальном API ФНС они есть, но API недоступен без токена. В scorer.py эти критерии
присутствуют и готовы к работе, просто никогда не срабатывают в текущей реализации.

**fns_api.py** — файл-заглушка для обратной совместимости: просто реэкспортирует `rusprofile_api`.

### 7.3. scorer.py — rule-based скоринг

5 критериев, каждый добавляет штрафные очки. Пороги хранятся в `config.py`.

| Критерий | Очки | Логика |
|---|---|---|
| Ликвидирована | 50 | Нельзя работать с несуществующей компанией |
| В процессе ликвидации | 40 | Высокий риск незавершённых сделок |
| Возраст < 3 месяцев | 30 | «Однодневки» обычно регистрируются незадолго до схемы |
| Возраст < 6 месяцев | 20 | — |
| Возраст < 12 месяцев | 10 | — |
| Массовый адрес | 20 | Один адрес на тысячи компаний = номинальный офис |
| Массовый директор | 20 | Один директор в сотнях компаний = номинал |
| Капитал ≤ 10 000 руб. | 10 | Минимально возможный уставный капитал |

Вердикт: ≤25 = `safe`, ≤60 = `caution`, >60 = `high_risk`.

### 7.4. UI — module2/ui.py

**_section_inn_input()**: поле ввода ИНН, валидация через `validator.py` прямо в форме.

**_run_check(inn)** — pipeline:
1. Проверить кэш PostgreSQL (`get_contractor_cache(session, inn)`)
2. Если кэш свежий (expires_at > now) — использовать кэш
3. Иначе — `get_entity_by_inn(inn)` → `score_contractor(data)`
4. Сохранить в кэш (`save_contractor(session, {...})`) с TTL 24 часа
5. Сохранить в `st.session_state["contractor_result"]`

**_section_result()** — карточка контрагента в стиле Т-Банка:
- Вердикт-карточка с цветной полосой (safe/caution/high_risk)
- Реквизиты: наименование, ИНН, ОГРН, статус, дата регистрации, адрес, капитал
- Сработавшие критерии с `border-left: 3px solid #FF3B30`
- Пометка «из кэша» если данные взяты из БД

## 8. ML-пайплайн

### 8.1. dataset.py — генерация синтетических данных

**Почему синтетика**: для обучения нужны размеченные данные (выписка + метка low/medium/high).
Реальных размеченных данных нет — это коммерческая тайна банков. Синтез по нормативам ЦБ РФ
является стандартной практикой в академических AML-исследованиях.

**Три стратегии генерации** (итого 10 000 строк):

**Стратегия A (65%) — Beta-распределения**:
Почему Beta, а не Gaussian: признаки ограничены [0, 1]. Гауссово распределение после
clip() накапливает вероятность у краёв (артефакт). Beta(a, b) — «родное» распределение
для долей и вероятностей, без артефактов.
Классы: low (среднее значение далеко от порога), medium (у порога), high (за порогом).
Латентный фактор интенсивности риска создаёт межпризнаковую корреляцию внутри класса:
если cash_ratio высокий — скорее всего высокий и fl_ratio (как в реальных схемах).

**Стратегия B (20%) — граничные случаи**:
Признаки генерируются случайно у самих порогов (±8% диапазона).
Метка присваивается правилом: 0-1 нарушений → low, 2-3 → medium, 4+ → high.
Создаёт трудные примеры: ИП с cash_ratio=0.32 при нормальных остальных признаках.
Без этих примеров модель была бы слишком уверена в граничных зонах.

**Стратегия C (15%) — отраслевые профили**:
9 конкретных бизнес-сценариев с вручную заданными параметрами:
- Розница/кафе: высокий cash_ratio (норма для общепита) → low
- ИТ-компания: высокая концентрация (2-3 заказчика норма) → low
- Строительство: много круглых сумм (сметы) → low
- Логистика: высокий transit_ratio (норма для перевозчиков) → medium
- Обналичивание через физлиц → high
- Транзитная схема → high
- Структурирование → high
- Замаскированная схема (некоторые признаки в норме намеренно) → high

**Результат**: 10 000 строк, распределение ~41% low / 25% medium / 34% high.
Дисбаланс компенсируется `class_weight="balanced"` в RandomForest.

### 8.2. train.py — обучение

```python
pipe = Pipeline([
    ("scaler", StandardScaler()),      # нормировка (RF не требует, но хорошая практика)
    ("clf",    RandomForestClassifier(
        class_weight="balanced",        # компенсация дисбаланса классов
        random_state=42,
    )),
])

param_grid = {
    "clf__n_estimators":     [100, 200, 500],
    "clf__max_depth":        [5, 10, None],
    "clf__min_samples_leaf": [1, 2, 5],
    "clf__max_features":     ["sqrt", "log2"],
}

cv = StratifiedKFold(n_splits=5, shuffle=True)
grid = GridSearchCV(pipe, param_grid, cv=cv, scoring="f1_macro", n_jobs=-1)
```

80/20 train/test split + 5-fold cross-validation на train-сете.
Лучшая модель сохраняется в `ml/model.joblib` через joblib.

**Метрики последнего обучения** (10 000 строк, Beta-датасет):
- Лучшие параметры: `max_depth=None, max_features=sqrt, min_samples_leaf=2, n_estimators=200`
- CV F1-macro: 0.9295
- Test accuracy: 0.94
- High recall: 0.97 (из 100 реально опасных выписок — 97 выявляются)
- Medium F1: 0.88 (medium/high иногда путаются — это реалистично)

**Почему 94% лучше, чем прежние 100%**: предыдущий датасет из нормальных распределений
был слишком легко разделим, модель переобучалась. 94% на данных с реалистичным перекрытием
классов означает более качественную генерализацию.

### 8.3. evaluate.py — отдельный скрипт оценки

```bash
python ml/evaluate.py
```

Загружает `model.joblib` и `data.csv`, печатает accuracy, F1-macro, F1 по классам, confusion matrix.
Не используется приложением — только для проверки качества модели разработчиком.

### 8.4. model.joblib — как используется в production

При первом вызове `predict()` в рамках Streamlit-сессии:
```python
_model = joblib.load("ml/model.joblib")  # загружается один раз
```
Объект хранится в модульной переменной `_model` — Python кэширует её между перезапусками
скрипта Streamlit (в рамках одного процесса). Таким образом, модель загружается один раз
при первом запросе и остаётся в памяти всё время работы сервера.

Синхронный движок (не async) — Streamlit работает синхронно, async здесь избыточен.

### 9.2. Три таблицы

**analysis_history** — каждый запуск анализа:
- `features_json` (JSONB) — все 10 признаков одним JSON-объектом
- `importances_json` (JSONB) — важности признаков из RF
- Связь 1:N с `risk_factors`

**risk_factors** — расшифровка по каждому признаку:
- `is_triggered` — превышен ли порог
- `importance` — вклад признака в решение модели
- ON DELETE CASCADE от analysis_history

**contractors** — кэш проверенных контрагентов:
- `expires_at` — время истечения кэша (24 часа по умолчанию из `CONTRACTOR_CACHE_TTL_HOURS`)
- `raw_json` — полный ответ от Rusprofile для отладки

### 9.3. crud.py

4 функции:
- `save_analysis(session, data)` — сохраняет AnalysisHistory + список RiskFactor
- `get_history(session, limit=50)` — последние N анализов
- `save_contractor(session, data, ttl_hours)` — upsert контрагента (INSERT ON CONFLICT UPDATE через merge)
- `get_contractor_cache(session, inn)` — возвращает запись если `expires_at > now`, иначе None

**Нюанс**: все вызовы БД в UI обёрнуты в `try/except` — если PostgreSQL недоступен,
приложение продолжает работать без истории и кэша.

## 12. Известные ограничения и нюансы

### Что не реализовано / упрощено

1. **mass_address / mass_director всегда False** — Rusprofile не предоставляет эту информацию.
   Официальный API ФНС требует токен. Критерии в scorer.py готовы, просто не срабатывают.

2. **Синтетические обучающие данные** — модель обучена на данных, сгенерированных по нормативам ЦБ РФ, а не на реальных банковских выписках. Для реального банка потребовались бы тысячи размеченных случаев.

3. **okved_mismatch без ОКВЭД = 0.0** — если пользователь не указал ОКВЭД, признак 4 всегда равен нулю и не влияет на классификацию.

4. **Rusprofile может быть недоступен** — сайт может заблокировать частые запросы или изменить структуру HTML. Система показывает ошибку и предлагает повторить.

5. **Нет аутентификации** — система предназначена для одного пользователя (ИП) или локального запуска.

6. **Нет сохранения файлов** — файлы выписок не хранятся на сервере, только вычисленные признаки. Это осознанное решение по безопасности.

### Технические нюансы Streamlit

- Каждое взаимодействие перезапускает весь Python-скрипт → важно кэшировать дорогие операции в `st.session_state`
- `st.rerun()` вызывается после завершения анализа — принудительно перерисовывает UI с новыми данными
- Модель ML загружается один раз в модульную переменную и не перезагружается между нажатиями

### Нюанс с импортами

`fns_api.py` — пустой файл-алиас для `rusprofile_api.py`. Сохранён потому, что в ранних версиях
кода был написан как `from module2.fns_api import get_entity_by_inn`. Переименование без удаления
алиаса сохраняет обратную совместимость.
